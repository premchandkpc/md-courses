Core Building Blocks
API Gateway · Load Balancer · Reverse Proxy
Three fundamental infrastructure components that live between clients and servers. Each solves a different problem — understanding when to use which is the most common interview trap.
Component Overview

API Gateway

Layer 7 · Managed Entry Point
A single, unified API entry point for a system. Routes requests to appropriate microservices, handles cross-cutting concerns like auth, rate limiting, and request transformation.
  • Request routing + versioning
  • Auth, rate limiting, logging
  • Protocol translation (REST↔gRPC)
  • Response aggregation (fan-in)
  • Single point of policy enforcement

Load Balancer

Layer 4/7 · Traffic Distributor
Distributes incoming requests across a pool of backend servers to maximize throughput, minimize response time, and avoid overloading any single server.
  • Health checks + failover
  • Session persistence (sticky sessions)
  • SSL termination
  • Algorithms: RR, least connections, IP hash
  • DDoS absorption at edge

Reverse Proxy

Layer 7 · Server-side Proxy
Sits in front of backend servers, intercepts requests from clients, and forwards them. Provides caching, compression, SSL termination, and hides backend topology.
  • Caching (static + dynamic)
  • Compression + TLS termination
  • Hides internal server topology
  • Request/response manipulation
  • Web application firewall (WAF)
Functional Requirements — What Each Does

Why This Matters

Interviewers ask "should I use an API Gateway or a Load Balancer?" to test if you understand the responsibility boundary. The answer is usually both — but at different layers. API Gateways handle application concerns (auth, routing, aggregation). Load balancers handle infrastructure concerns (distribution, health, failover). Reverse proxies handle performance concerns (caching, compression, termination).
