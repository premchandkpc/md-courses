🤖 LLM Serving Architecture

System Overview

🎯 Core Challenge

Serve 70B parameter models with <500ms TTFT at billion-scale

Billions of users. Cost per token critical. Safety constraints.

Token Generation Pipeline

Standard (Non-Batched): User 1: Generate 1 token (100ms) User 2: Wait for User 1... Sequential = SLOW Batched (N=100): User 1-100: Batch 100 tokens together Latency per user: 100ms (parallel!) Throughput: 100x better Token generation = compute bottleneck Batching hides latency

Component 1: Request Batching

Accumulation Strategy

Batch size:100 concurrent users
Accumulation time:10ms
First-token latency:~110ms
Throughput:1000 tokens/sec per GPU

Component 2: KV Cache Optimization

Without KV Cache: ├─ For each new token ├─ Recompute attention for ALL previous tokens ├─ Expensive: O(n^2) with sequence length With KV Cache: ├─ Store K,V from previous tokens ├─ Reuse in next generation ├─ Only compute new token attention ├─ 10x faster token generation (100ms → 10ms) Trade-off: ├─ Memory growth: ~2.5GB per 1000-token context ├─ Acceptable: speed gain worth it └─ Total latency: 1000 tokens in 10 seconds

Component 3: Speculative Decoding

Parallel Token Generation

  • Main model: Generates token i (100ms)
  • While computing: Drafter model guesses tokens i+1, i+2, i+3 (20ms)
  • Verify: Check if drafter matched main model
  • Result: 2-3x speedup (if guesses correct)

Component 4: Token-by-Token Streaming

User Experience

Traditional: Wait 3 seconds → see complete response (feels slow)

Streaming: See token 1 at 100ms, token 2 at 200ms, etc. (feels fast)

  • Technology: WebSocket or Server-Sent Events (SSE)
  • Perceived latency: First token (100ms)
  • Progressive rendering on client

Component 5: Model Management

Multi-Model Serving

Small (7B):10ms/token, cheap
Medium (70B):100ms/token, balanced
Large (540B):500ms/token, powerful

Router Logic

Classify query complexity → route to appropriate model

Simple queries: Use small model (faster, cheaper)

Complex reasoning: Use large model (better quality)

Component 6: Inference Optimization

Quantization

float32 → int8: 4x smaller, 4x faster, 0.5% accuracy loss

Distillation

Train 10x smaller student on teacher model: 10x faster inference on CPU

Request Scheduling

Priority queue: Paying users first, manage backpressure

Safety & Cost Control

Safety Filtering

  • Prompt check: Block harmful requests early
  • Output check: Detect toxic content in response
  • Latency: <50ms (separate GPU)

Token Limits

  • Free users: Max 500 tokens/request
  • Paid users: Max 2000 tokens/request
  • Enforcement: Stop generation at limit

Monitoring & Scaling

Metrics

  • Time-to-first-token (TTFT)
  • Tokens-per-second (TPS)
  • Cost-per-token
  • Queue depth (backpressure signal)

Scaling 10x

Optimizations: Quantization + larger batches + distillation

Result: Handle 10x load with 3-5x GPU increase (not 10x)
