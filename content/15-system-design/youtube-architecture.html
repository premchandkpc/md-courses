🎯 Core Challenge

Rank 1B+ videos for 2.5B users in <500ms

80% of all watch time comes from recommendations. Every millisecond and every ranking decision impacts billions of view hours.

Component 1: Candidate Generation

Retrieve ~1000 potential videos from 1B+ corpus in <100ms using multiple strategies.

Collaborative Filtering

Find similar users via embedding similarity, aggregate their watched videos.

Candidates: 300
Index: FAISS (user-user)
Latency: <30ms

Content-Based

Find videos similar to user's watch history using semantic embeddings.

Candidates: 400
Index: FAISS (video-video)
Latency: <30ms

Trending & Fresh

Surface popular and newly-uploaded videos for discovery.

Candidates: 300
Source: Manually curated
Latency: <10ms (cached)

📊 FAISS Index Strategy

  • Index Type: IVF (Inverted File) with ~1000 clusters
  • Recall: 90% (good balance of speed vs accuracy)
  • Storage: 1 TB for video embeddings (100M hot videos in memory)
  • Update Frequency: Hourly recomputation
  • Sharding: 1000 shards by video_id % 1000

Component 2: Ranking Service

Score and order candidates using deep neural network predicting watch-time.

🤖 Model Architecture: Two-Tower Neural Network

User Tower: Video Tower: Input (100D) Input (100D) │ │ ▼ ▼ Dense(512) + ReLU Dense(512) + ReLU │ │ ▼ ▼ Dense(256) + ReLU Dense(256) + ReLU │ │ ▼ ▼ Dense(128) + ReLU Dense(128) + ReLU │ │ └────────────┬───────────┘ │ ▼ Dot Product │ ▼ Dense(1) + Sigmoid │ ▼ Watch-Time Prediction (0-360 mins)

Input Features (200 total)

User Features (100)

  • Watch history embedding (256D)
  • Average watch duration
  • Preferred categories
  • Country, language, device
  • Time of day bias

Output & Optimization

Inference Latency

  • Batch size: 1000 candidates
  • GPU latency: 10ms per batch
  • Throughput: 100K rankings/sec
  • GPUs needed: 10K+ (distributed)

Multi-Objective Ranking Formula

📐 Weighted Score Computation

score = 0.50 * watch_time_pred + 0.15 * freshness_boost + 0.10 * creator_quality_score + 0.15 * engagement_signals + 0.10 * diversity_penalty + 0.05 * exploration_bonus Freshness boost: IF video_age < 24h: +0.2 to score IF video_age < 1h: +0.5 to score Trends faster than old content Diversity penalty: Reduce similar videos in same category MAX 3 videos per category in top 20 Exploration bonus (Thompson sampling): New videos get small boost (~2-5%) Learn true CTR from exploration Gradually shift to exploitation

Component 3: Re-ranking & Diversity

Deduplication

Remove videos user has already watched or interacted with.

  • Check watch history (Redis)
  • Remove liked videos
  • Skip recently shown

Diversity Filtering

Ensure variety in categories, creators, content types.

  • MAX N per creator
  • MAX N per category
  • Vary duration

Freshness Boost

Prioritize trending and newly-uploaded content.

  • New uploads → top ranks
  • Trending videos → boost
  • Creator livestreams → highest

⚡ Latency: <5ms (mostly filtering, no ML)

Feature Store Architecture

Real-Time Feature Serving

  • Hot Tier (Redis): 400 GB user features + 500 GB video features
  • Warm Tier (Disk): Full feature database (terabytes)
  • Update Frequency: Streaming updates from Kafka every 1 minute
  • Cache Hit Rate: >95% (most features cached)
  • Latency SLA: <5ms for 99th percentile

Sharding Strategy

User features: By user_id
Video features: By video_id
Shards: 1000 per type
Replication: 3x

Update Pipeline

Source: Kafka streams
Processing: Flink
Write back: Redis publish
Latency: <1 min

Scaling Strategy: 100x Traffic

Current: 350K QPS | Scaled: 35M QPS

  • Quantization: fp32 → int8 (4x compression, 4x faster)
  • Batch size increase: 1000 → 10K (10x throughput)
  • Model distillation: 10x smaller student model on CPU
  • Approximate ranking: Rank only top 100 instead of 1000

Result: Handle 35M QPS with 1000 GPUs (vs 10K without optimization)
