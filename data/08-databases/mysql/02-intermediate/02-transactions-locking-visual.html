Deadlock Patterns: Circular Wait Visualization
Shows how circular dependencies cause deadlocks. Transaction A locks row 1, waits for row 2. Transaction B locks row 2, waits for row 1. Solution: lock ordering (always lock in same order prevents cycles). MySQL detects deadlock within 50-100ms, rolls back one transaction.

Lock Types: Shared vs Exclusive vs Gap
Compares three lock types. Shared locks (S): multiple readers, one writer blocks. Exclusive locks (X): single reader/writer, all others blocked. Gap locks: prevent inserts in range (prevent phantoms). Gap lock cost: 5-15% latency overhead but prevents phantom reads in REPEATABLE READ.
Lock Type Multiple Readers Writer Blocked Purpose Typical Usage
Shared (S) ✓ Yes ✓ Yes Consistent reads SELECT statements (read-only)
Exclusive (X) ✗ No ✗ No Write protection UPDATE/DELETE/INSERT
Gap Lock ✓ Readers ok ⚠ Inserts blocked Phantom prevention Range queries (REPEATABLE READ)
Intent Lock (Table) ✓ Readers ok ⚠ ALTERs blocked Prevent schema changes Coordination during DDL
Undo Log Growth: Transaction Duration vs Memory
Shows how long-running transactions cause undo log to grow. Short transactions (10ms): minimal undo log (100KB). Normal transactions (100ms): moderate growth (1MB). Long reports (10 minutes): massive growth (5-50GB). Undo log growth slows queries (must search for historical versions). Solution: keep transactions short (<100ms, <1000 rows modified).

Recovery Time: Database Size vs Restart Duration
Shows time to recover database after crash. Small database (100GB): 30-60 seconds (replay redo log). Large database (1TB): 120-300 seconds. Huge database (10TB): 500-1200 seconds. Recovery time depends on redo log replay speed, not data size. Faster storage (SSD): 2-3x faster recovery. Slow storage (HDD): slower recovery.

Real-World Transaction Scenarios
Three production examples showing isolation level choice, locking strategy, timeout configuration, and lessons learned. Each represents different consistency vs performance requirements.
🛒 Scenario 1: E-Commerce Cart (High Traffic, Speed Critical)
Isolation Level: READ COMMITTED
Why: Cart is eventually consistent (user adds item, might see old version briefly). Stronger isolation (REPEATABLE READ) = 3-5x slower = worse UX. Accept stale reads (user refreshes page if needed).
Lock Timeout: 5 seconds (fast failure detection)
Typical Latency: 10-50ms per cart operation
Peak Throughput: 50K cart updates/sec across 8-core database
Deadlock Handling: Retry 3x with 10ms backoff (rare, <1 per 1000 ops)
✓ Success: Fast, users happy, deadlocks rare, no anomalies matter
💰 Scenario 2: Banking Transfer System (Compliance Critical)
Isolation Level: SERIALIZABLE
Why: Legal requirement: all transfers must be atomic + isolated. Cannot have dirty reads (account shows money not transferred). Cannot have phantom reads (transaction isolation audits). Compliance > performance.
Lock Strategy: Lock ordering enforced (sort account IDs before updating)
Lock Timeout: 10 seconds (stricter timeout due to gap locks)
Typical Latency: 100-500ms per transfer (gap locks add overhead)
Peak Throughput: 100-200 transfers/sec (much lower due to serializable)
Deadlock Rate: <1 per 10K transfers (lock ordering prevents most)
✓ Correct: Every transfer auditable, fully isolated, compliant
📦 Scenario 3: Inventory System (Prevent Oversell)
Isolation Level: REPEATABLE READ
Why: Must prevent phantom stock additions (gap locks prevent new stock rows). Must prevent overselling (repeatable reads ensure consistent stock count). MySQL default balances both.
Locking Strategy: SELECT ... FOR UPDATE on stock row (explicit lock)
Lock Timeout: 3 seconds (inventory ops should be fast)
Typical Latency: 50-200ms per reservation
Peak Throughput: 500 reservations/sec (gap locks prevent race conditions)
Gap Lock Behavior: Prevents new stock row inserts during reservation
⚠ Trade-off: 5-10% latency overhead from gap locks, but prevents oversell
