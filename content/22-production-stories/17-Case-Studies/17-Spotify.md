# 17-Spotify

Spotify's engineering culture is famous for the squad/tribe/guild organizational model. Their backend architecture evolved from a monolithic Ruby-on-Rails app to a microservices ecosystem, with interesting patterns in music discovery, client-server architecture, and offline playback.

## Problem
Spotify started as a desktop-only music streaming app with a Ruby-on-Rails backend. As they grew to 100M+ users, expanded to mobile, added podcast support, and entered new markets, the monolith became a bottleneck. They needed an architecture that supported rapid experimentation (A/B testing everything), offline playback, and personalized discovery at scale.

## Architecture
- **Squad/Tribe/Guild Model**: Squads (6-12 people) are autonomous teams owning a feature area. Squads are grouped into Tribes (aligned with a business domain). Chapters (same-role groups across squads) and Guilds (communities of interest) enable cross-team knowledge sharing. This model allows Spotify to scale while maintaining startup-like autonomy.
- **Backend Services (200+ services)**: Services are organized around business capabilities — playlist management, music metadata, search, recommendations, ads, subscriptions, social features, and audio delivery. Each service has clear ownership, including on-call responsibility.
- **Music Discovery System**: Spotify's recommendation engine uses collaborative filtering (what users with similar tastes listen to), natural language processing (analyzing blogs, reviews), and audio analysis (tempo, key, loudness, genre classification). Daily Mixes, Discover Weekly, and Release Radar are generated by combining these signals.
- **Client-Server Architecture (Smodule)**: Spotify's client uses a plugin system called Smodule — each feature is a self-contained plugin that communicates with a corresponding backend service. Plugins are isolated (a crash in one doesn't affect others) and can be updated independently of the client application.
- **Offline-first**: Music is downloaded to the device for offline playback. The client manages sync state and resolves conflicts when the device comes back online (play count updates, playlist modifications). Offline mode is a first-class feature, not an afterthought.
- **Pub/Sub for real-time features**: Social features (friend activity, collaborative playlists) use a publish-subscribe system. When a friend starts playing a song, the event is published and all subscribed clients receive a push notification.

## Lessons Learned
- **Organizational structure drives architecture**: The squad model works because squad boundaries align with service boundaries. If squads need to coordinate on every change, the model breaks. Ensure squads can deploy independently — the code and team boundaries must match.
- **Client-side architecture matters**: Spotify invested heavily in client architecture (Smodule, offline sync, streaming logic). For consumer products, the client is as important as the backend. Don't neglect client-side architecture when designing your system.
- **Experimentation is a platform**: Spotify A/B tests everything — UI changes, algorithm changes, feature rollouts. The experimentation platform is a first-class infrastructure component. Non-trivial product decisions are backed by statistically significant experiments.
- **Run your own infra when you need flexibility**: Spotify ran their own data centers for years before migrating to Google Cloud. They built custom infrastructure for audio streaming and recommendations. Choose build vs buy based on what's core to your competitive advantage.

## Mental Model
Spotify's squad model is like a jazz ensemble rather than a classical orchestra. Classical orchestras have a conductor (top-down) who directs every musician. Jazz bands have small groups (squads) that improvise independently but harmonize through a shared understanding of the music (platform APIs, cultural norms). The result is faster creativity, more ownership, and still a coherent performance.
