PostgreSQL Advanced Optimization & Performance Tuning

Query Execution Insights

EXPLAIN ANALYZE

Shows actual execution stats with real row counts and timing. Compare planned vs actual to catch statistics mismatches.

Essential for diagnosis

Cost Interpretation

Format: (startup..total). Seq Scan ~1000 = slow. Index Scan ~10 = fast. Relative not absolute metric.

Plan Types

  • Seq Scan: Full table (bad)
  • Index Scan: Use index (good)
  • Nested Loop: Expensive
  • Hash Join: Better
Index Strategy by Type

B-tree (Default)

Most common. Fast for range queries, equality, sorting. Use for normal columns.

General purpose

BRIN (Range)

For huge tables (1B+ rows) with sequential data. Minimal storage overhead. 100x+ faster than full scan.

Time-series, logs

GIN (Inverted)

Best for JSON, arrays, full-text search. Slower writes, faster reads for complex queries.

JSON, arrays, FTS

Hash

Equality only, no range. Rarely used. GiST/BRIN better alternatives.

Niche use

GiST (Search Tree)

Geometric, full-text search, IP ranges. Flexible but slower than specialized indexes.

Geometric, GIS

SP-GiST (Space-partitioned)

IP addresses, ranges, spatial. Better than GiST for hierarchical data.

IP ranges, spatial

Configuration Tuning Checklist

Memory Settings

  • shared_buffers: 25% RAM
  • effective_cache_size: 75% RAM
  • work_mem: Per sort op
  • maintenance_work_mem: REINDEX, VACUUM

Query Planning

  • max_parallel_workers: Enable parallelism
  • random_page_cost: SSD=1.1, HDD=4.0
  • enable_*: Control joins, scans

Maintenance

  • VACUUM: Reclaim space
  • ANALYZE: Update stats
  • Autovacuum: Automatic cleanup

Real-World Optimization Example

User Post Count (Materialized View)

Slow Query: JOIN users + posts + GROUP BY + ORDER BY

-- Before: 2-5 seconds SELECT u.id, COUNT(p.id) as post_count FROM users u LEFT JOIN posts p GROUP BY u.id ORDER BY post_count DESC;

Solution: Materialized view + index

CREATE MATERIALIZED VIEW user_stats AS SELECT u.id, u.email, COUNT(p.id) as post_count FROM users u LEFT JOIN posts p ON u.id=p.user_id GROUP BY u.id; CREATE INDEX idx_user_stats_count ON user_stats(post_count DESC);

After: <10ms (sub-second refresh)

Monitoring & Diagnostics

pg_stat_statements

Built-in query profiler. Find slow queries, execution count, timing stats.

SELECT query, mean_time FROM pg_stat_statements ORDER BY mean_time DESC;

pg_stat_user_indexes

Index usage tracking. Identify unused indexes (remove to speed writes).

SELECT indexname, idx_scan FROM pg_stat_user_indexes;

Key Takeaways

Always Profile First

Run EXPLAIN ANALYZE before optimizing. Real bottlenecks often surprise you.

Index Strategy Matters

B-tree for most. BRIN for huge tables. GIN for JSON/arrays. Wrong index = wasted time.

Configuration is Foundation

Tuning memory/workers/costs provides baseline improvement before schema changes.

Monitor Continuously

Slow queries change over time. Regular profiling catches new bottlenecks early.
