Query vs Scan

✓ Query (Fast)

Use partition key + optional sort key

Retrieves items from single partition.

KeyCondition: userId = 'user123' AND timestamp BETWEEN '2026-01-01' AND '2026-01-31'

⚡ Best for app queries

✗ Scan (Slow)

Read all items in table

Reads every partition sequentially.

Scan entire table Filter by status='active' (Still costs for all items!)

🔥 Use only for import/export

💡 Key Difference

Query: RCU = items scanned / 4KB (usually small)

Scan: RCU = all table items scanned / 4KB (expensive!)

Query Flow with Pagination

Step 1: Query with Limit=10
Gets first 10 items matching key condition
Step 2: Check LastEvaluatedKey
If exists: more results available
Step 3: Query again with ExclusiveStartKey
Continue from where previous left off
Step 4: Repeat until LastEvaluatedKey is null
All results fetched

Performance Optimization

Read Consistency Use eventually consistent (default) → 50% cheaper
Query Over Scan Always use partition key → much faster
Projection Include only needed attributes
Batching BatchGetItem/WriteItem → 33% cheaper
Conditional Writes Fail early, avoid unnecessary updates
Connection Pooling Reuse SDK clients across requests
