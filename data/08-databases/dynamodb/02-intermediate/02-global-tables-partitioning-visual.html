Hot Partition Problem: Writes Concentration
Visualizes how poor partition keys concentrate writes on single partitions. Sequential keys (timestamp) cause all writes to hit the same partition until time advances. Good partition keys (UUID, user_id) distribute writes evenly. Hot partitions throttle at 40KB/sec and cause 5-30 minute recovery delays.

Global Secondary Index (GSI) Trade-offs
GSI provides alternative query path but with independent partition key and throughput limits. Main table and GSI can have different hot partition patterns. GSI queries are always eventually consistent. Throughput provisioning is separate (costly).

Real-World Partition Key Scenarios
Three production examples showing partition key choices, partition counts, cost analysis, and lessons learned. Each represents different scaling challenges and solutions.
🚗 Scenario 1: Ride-Sharing (50M Active Trips)
Partition Key: {driver_id, trip_timestamp}
Why: Naive approach (trip_timestamp alone) = hot partition (all new trips write to latest timestamp). Solution: hash driver_id into partition key. Per-driver writes distribute across thousands of partitions.
Table Size: 500GB (50M trips × 10KB avg)
Partition Count: 50 partitions (500GB ÷ 10GB)
Write Rate: 50K trips/sec peak evenly distributed
Provisioned WCU: 2,000 WCU (50K per sec ÷ 25 = 2000)
Query Pattern: Driver trip history (range on trip_timestamp) = targeted query, fast
✓ Success: Partition balance maintained, no throttling under peak load
💬 Scenario 2: Messaging (1B Active Conversations)
Partition Key: {conversation_id, message_timestamp}
Challenge: Popular conversations (celebrity group chats, 1M messages/hour) create hot partitions. Normal conversation = 10 messages/hour (cold). Uneven load.
Solution: Shard hot conversations at app layer. conversation_id={conv_id}#{shard_0..9} distributes popular convos across 10 partition keys.
Table Size: 1TB (1B conversations × 1KB metadata)
Partition Count: 100 partitions + app-level sharding for hot conversations
Write Rate: 100K messages/sec average, 500K/sec during viral moments
Provisioned WCU: 20,000 WCU (provisioned for 500K peak)
⚠ Trade-off: High WCU cost, but on-demand option available (pay per write, no provisioning)
📊 Scenario 3: Analytics Events (100B Events/Day)
Partition Key: {tenant_id, event_bucket_6hour}
Why: Time-series data naturally wants sequential write pattern. Solution: bucket into 6-hour windows instead of exact timestamp. tenant_id provides multi-tenancy isolation.
Write Pattern:
  • Large tenant: 100K events/sec
  • Medium tenant: 10K events/sec
  • Small tenant: 100 events/sec
Partitions per Bucket: ~20 partitions active per 6-hour window
Provisioned WCU: 5,000 WCU (per bucket window)
Rolling Retention: Drop buckets older than 90 days (no delete operations, just drop old partition keys)
✓ Optimized: Time-series bucketing avoids sequential hot partition problem
