Eviction Policy Impact on Cache Hit Rate
Shows cache hit rate under different eviction policies. LFU (least-frequently-used) achieves highest hit rates (85-98%) by evicting rarely-accessed keys. LRU (least-recently-used) is good (80-95%) for temporal locality. Random and NoEviction are less predictable. Better hit rate = fewer database calls = lower latency.

Memory Pressure & Eviction Latency Spikes
Shows how memory utilization affects write latency. Below 80% capacity, no eviction occurs and latency stays low (1-2µs). As capacity increases to 80-95%, eviction kicks in (5-50ms spikes every ~100 writes). Above 95%, constant eviction causes unacceptable latency (100-500ms). Optimal range: keep Redis at 60-80% capacity.

TTL vs Eviction: Cleanup Strategy Trade-offs
Shows trade-offs between TTL (explicit expiration) and eviction-based cleanup. Pure TTL requires periodic scanning (~10% CPU). Pure eviction handles cleanup automatically but may evict valuable data. Hybrid approach: set TTL on important data + rely on eviction for overflow.
Strategy CPU Overhead Memory Fragmentation Data Loss Risk Best For
Pure TTL 8-15% High None Sessions, auth tokens
Pure Eviction 1-2% Low High Bounded caches
Hybrid (Both) 3-5% Low Low Production systems
Multi-Tier Caching Architecture
Shows three-tier caching pyramid. L1 (local process cache) provides fastest access (1-2ms) but limited size (100MB-1GB). L2 (Redis) trades network latency (5-15ms) for shared capacity (1GB-100GB). L3 (Database) is authoritative source, slowest (50-500ms) but unlimited. Each tier reduces load on lower tiers, improving overall system performance.

Hot Key Problem: Single Key Bottleneck
Shows how popular keys create single-node throughput bottleneck. A key accessed 100K times/sec concentrates all requests on one Redis connection, capping throughput at ~50K ops/sec. Solutions: local replica on each client (stale but fast), or Redis Cluster with key redistribution. Trade-off: staleness vs throughput.

Pipeline vs Transaction: Throughput vs Atomicity Trade-off
Shows performance trade-off between pipelining (fire-and-forget, non-atomic) and transactions (MULTI/EXEC, atomic but slower). Pipelining achieves 200K+ ops/sec but has no atomicity. Transactions guarantee atomicity (all-or-nothing) but add latency from blocking. Lua scripts provide best of both: atomic without MULTI overhead.
Pattern Throughput Latency Atomicity Use Case
Fire & Forget 200K+ ops/sec 1-2µs None Metrics, logs (loss ok)
Pipeline 150K ops/sec 5-10µs Per-command Batch operations
Transaction (MULTI) 50K ops/sec 50-200µs Full Critical updates
Lua Script 100K ops/sec 10-50µs Full Complex operations
Real-World Redis Deployment Scenarios
Three production examples showing cache sizing, eviction policies, memory management, and lessons learned. Each represents different access patterns and scaling challenges.
🔑 Scenario 1: Session Cache (100M Active Sessions)
Data Per Session: 1KB average (user_id, token, metadata)
Total Memory Required: 100GB (100M sessions × 1KB)
Eviction Policy: LRU (allkeys-lru) - oldest sessions evicted first
TTL Strategy: 30-minute expiration (explicit + eviction as backup)
Cluster Config: 10 nodes × 10GB each = 100GB total
Expected Hit Rate: 92% (users return frequently)
Memory Pressure: Keep at ~70% capacity (70GB used, 30GB headroom)
✓ Success: LRU evicts inactive users, TTL handles explicit expiry, no cold start problem
🏆 Scenario 2: Leaderboard Cache (1B Scores)
Data Structure: Redis Sorted Sets (player_id → score)
Memory Per Score: 24 bytes (id + score + metadata)
Total Size (all): 24GB (1B × 24 bytes)
Hot Subset: Top 1M scores accessed 90% of the time = 24MB hot
Eviction Policy: LFU - keep frequently-ranked players in cache
Cluster Config: 8 nodes × 3GB hot subset + overflow to disk
Query Pattern: Top 100 = 50ms (cached), Random player = 200ms (DB + cache)
⚠ Challenge: Rank changes trigger cache invalidation (complex coordination)
📡 Scenario 3: API Response Cache (50B Responses/Day)
Response Size: 200 bytes average (typical API JSON)
Cache Key: hash(endpoint + params)
Cardinality: 100M unique cache keys (many param combinations)
Hot Working Set: 20% of keys = 20M keys, 4GB memory
TTL: 5 minutes (balance between freshness and cache efficiency)
Memory Budget: 50GB Redis cluster (covers 250M responses)
Expected Hit Rate: 75-80% (some users request same data)
✓ Optimized: 5-min TTL prevents stale data, LRU eviction handles tail requests, local cache (app-side) for request-local hits
