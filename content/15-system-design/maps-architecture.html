🗺️ Google Maps Ranking Architecture

System Overview

User Search: "coffee near me" │ ▼ Query Understanding │ ▼ Geographic Retrieval (find nearby) │ ▼ Ranking (relevance + distance + popularity) │ ▼ Diversity Filter │ ▼ Top Places + Map

🎯 Core Challenge

Rank 200M places by relevance + distance + popularity at scale

Geographically distributed. Real-time signals. Regional spikes.

Scale & Metrics

Place Index

  • 200M+ places indexed
  • 100K+ QPS
  • <200ms p99 latency
  • Geographically distributed serving

Component 1: Geospatial Retrieval

Hilbert Curve Indexing

  • Space-filling curve: Converts 2D coordinates to 1D index
  • Property: Nearby places → nearby curve values
  • Range query: Find places in bounding box
  • Latency: <10ms geographic retrieval
Geographic Retrieval: Input: (user_lat, user_lng, search_radius) 1. Find bounding box 2. Hilbert curve range query 3. Return nearby places 4. Latency: <10ms

Component 2: Ranking Signals

Multi-Signal Ranking

score = w1 * relevance_score + w2 * distance_score + w3 * popularity_score + w4 * freshness_score where: relevance: query-place text match (BM25) distance: 1 / (1 + distance_km) popularity: ratings * review_count freshness: hours_since_update

Real-Time Signals (Updated Minutely)

  • Popularity (from user visits)
  • Crowdedness (estimated wait)
  • Recent reviews & ratings
  • Current hours (open/closed)

Component 3: Learning-to-Rank

Model

  • Type: Gradient Boosted Decision Trees (GBDT)
  • Training: Click logs from Maps searches
  • Features: Relevance + distance + popularity
  • Output: Ranking score

Component 4: Geographic Distribution

Problem

User in Singapore shouldn't wait for index in US datacenter

Solution: Geo-Partitioned Indexes

  • Split world: Into geographic regions
  • Each region: Has full index
  • Routing: Route requests to nearest region
  • Replication: 3x across datacenters per region
  • Latency: <50ms from nearest region

Regional Spikes

Rush hour: More queries in peak areas

Solution: Load balancing + cached results fallback

Component 5: Personalization

User Preferences

Same search, different results:

  • Vegetarian user: Rank vegetarian restaurants higher
  • Budget-conscious: Rank cheap places higher
  • Frequent visitor: Rank favorites higher

Implementation

Personalization features: User preferences + search history

Re-rank base results with personalization signal

Special Challenges

Region Boundaries

User at border (between two regions):

  • Query both regions
  • Merge results
  • Return top-K across regions

Real-Time Updates

When place status changes (opens/closes/updates):

  • Update feature in all regional indexes
  • Propagate changes (seconds to minutes)
  • Consistency eventual (acceptable)

Monitoring & Optimization

Key Metrics

  • Click-through rate on first result
  • User choosing different ranking
  • Regional latency variance
  • Place data freshness

Scaling 10x

Solutions: More regional indexes + caching + approximate ranking

Challenge: Keeping indexes fresh across globe
