# Service Mesh

A service mesh is a dedicated infrastructure layer that handles all service-to-service communication in a microservices deployment. It offloads networking concerns — traffic management, security, observability — from application code to a proxy sidecar. The mesh consists of a data plane (sidecar proxies that intercept all traffic) and a control plane (management component that configures the proxies).

## Overview

In a service mesh, every service instance has a co-located proxy (sidecar) that intercepts all incoming and outgoing traffic. These proxies form the data plane. A central control plane distributes configuration (routing rules, security policies, telemetry config) to all proxies. The application code remains unaware of the mesh — it sends plain HTTP/gRPC to localhost, and the sidecar handles mTLS, load balancing, retries, circuit breaking, and telemetry. Istio (Envoy-based) and Linkerd (Rust-based) are the dominant implementations.

## Key Characteristics

- **Traffic Management**: Fine-grained control over service-to-service traffic: request routing (version-based, header-based), traffic splitting (canary, blue-green), load balancing (round robin, least request, ring hash), and fault injection (delays, aborts). Rules are configured on the control plane and pushed to all sidecars.
- **Security**: Automatic mTLS between all mesh services. The control plane issues and rotates certificates. Authorization policies define which services can communicate. The mesh encrypts all east-west traffic without any application changes. Certificate rotation is transparent and zero-downtime.
- **Observability**: Distributed tracing, metrics (HTTP request count, latency, error rate, success rate), and access logs are generated by the sidecars and exported to backends (Prometheus, Jaeger, Grafana, OpenTelemetry). Each proxy produces detailed telemetry for every request flowing through it, providing full mesh visibility.
- **Resilience**: Circuit breakers stop traffic to failing services. Retries (with timeout and budget) handle transient failures. Timeouts prevent cascading delays. Health checking detects unresponsive instances. The mesh enforces these patterns consistently across all services rather than relying on each service to implement them.
- **Protocol Support**: HTTP/1.1, HTTP/2, gRPC, and TCP protocols are supported. The mesh can transparently upgrade protocols (HTTP/1.1 to HTTP/2). gRPC load balancing works correctly — a notoriously difficult problem with standard Kubernetes Services — because the sidecar performs L7 load balancing.
- **Platform Independence**: The mesh operates at the infrastructure level. Applications can be written in any language and use any framework. The service mesh does not require language-specific libraries or SDKs. This is ideal for polyglot microservices environments.

## Why It Matters

As microservices proliferate, the operational burden of networking, security, and observability shifts from per-service implementation to infrastructure. The service mesh provides these capabilities consistently, correctly, and without application changes. Teams can add mTLS, distributed tracing, and traffic splitting to an existing mesh of services by simply deploying a sidecar — no code changes required.

## Mental Model

A service mesh is like air traffic control for microservices. Each airplane (service) has a transponder (sidecar proxy) that reports its position, speed, and destination. The control tower (control plane) sees all flights, routes them safely, and enforces rules (altitude limits, no-fly zones). Pilots (application developers) focus on flying the plane — the tower handles separation, collision avoidance, and flight tracking. Adding a new airplane is as simple as installing its transponder.
