Single vs Composite Keys

Single Partition Key

Use: One item per partition key value

PK: userId - user123 - user456 - user789

Best for: Unique records, user profiles, singleton data

Partition + Sort Key

Use: Multiple items per partition, ordered access

PK: userId | SK: timestamp - user123 | 2026-01-01 - user123 | 2026-01-02 - user456 | 2026-01-01

Best for: Time series, posts, messages, versions

Design Anti-Patterns

❌ Timestamps as PK

All writes today → same partition → hot partition & throttling

Problem: Partition overloaded, requests throttled

❌ Sequential IDs

userId 1, 2, 3... → writes cluster together → hot partition

Problem: New users hit same partition first

❌ Low Cardinality

Only 3 status values but millions of records

Problem: One partition gets 99% of traffic

Global vs Local Secondary Indexes

Main Table

  • PK: userId
  • Query by userId
  • Primary data

GSI: Email Index

  • PK: email
  • Query by email
  • Separate throughput

LSI: Date Index

  • PK: userId
  • SK: createdDate
  • Created at table time

Capacity Planning

Read Capacity Unit (RCU)

1 RCU = 1 × 4KB read per second (strongly consistent)

OR 2 × 4KB reads per second (eventually consistent)

Read 1KB item → 1 RCU Read 5KB item → 2 RCU (rounded up) Read 8KB item → 2 RCU Read 9KB item → 3 RCU

Write Capacity Unit (WCU)

1 WCU = 1 × 1KB write per second

Write 1KB item → 1 WCU Write 2.5KB item → 3 WCU (rounded up) Write 5KB item → 5 WCU

Example: E-commerce Site

Workload: 1000 reads/sec, 100 writes/sec, ~2KB average item

RCU: 1000 reads × 1 RCU = 1000 RCU WCU: 100 writes × 2 WCU = 200 WCU Monthly cost: ~$3,000 (Provisioned, us-east-1)
