# Production Failure: Hydration Mismatch

## WHAT
Hydration mismatch occurs when the HTML generated by SSR doesn't match what React expects on the client. React detects the mismatch during hydration and **discards the server HTML**, replacing it with client-rendered content — causing a full re-render (and often, a flash).

## WHY
```
Server renders:  <div>Welcome, User</div>
Client renders:  <div>Welcome, admin</div>
                  ↑ Mismatch! React throws warning, re-renders entire tree
```

Root causes:
- Browser-specific APIs (`window.innerWidth` used in render)
- Dynamic data from cookies/localStorage
- Time/date formatting differences (server vs client timezone)
- Third-party scripts that modify DOM before hydration
- CSS-in-JS with different server/client styles

## INTERNALS

### The Hydration Algorithm

```typescript
function hydrateInstance(
  instance: Instance,
  type: string,
  props: Props,
  rootContainerInstance: Container,
) {
  // Check if server-rendered DOM matches expected
  const expectedProps = prepareUpdate(instance, type, oldProps, props);
  
  if (expectedProps) {
    // Mismatch! React logs warning and re-renders
    console.error(
      'Hydration failed because the initial UI does not match ' +
      'what was rendered on the server.'
    );
    
    // React forces a client-side re-render
    // This discards ALL server HTML
    throw new Error('Hydration mismatch');
  }
}
```

When hydration fails, React:
1. Logs a warning (dev) or error (prod with `onRecoverableError`)
2. Deletes the entire server-rendered DOM subtree
3. Re-renders from scratch on the client
4. This causes **layout shift, flash, and wasted work**

## COMMON FAILURE PATTERNS

### 1. `window` / `document` access during render

```typescript
// ❌ BROKEN: window is undefined on server
function WidthDisplay() {
  const width = window.innerWidth; // Throws on server!
  return <div>{width}px</div>;
}

// ✅ FIXED: useEffect runs only on client
function WidthDisplay() {
  const [width, setWidth] = useState(0);
  useEffect(() => {
    setWidth(window.innerWidth);
    const handler = () => setWidth(window.innerWidth);
    window.addEventListener('resize', handler);
    return () => window.removeEventListener('resize', handler);
  }, []);
  return <div>{width}px</div>;
}
```

### 2. Timestamps in render

```typescript
// ❌ BROKEN: Different timezone on server vs client
function TimeDisplay() {
  return <div>{new Date().toLocaleString()}</div>;
}

// ✅ FIXED: Consistent timestamp
function TimeDisplay() {
  const [time, setTime] = useState('');
  useEffect(() => {
    setTime(new Date().toLocaleString());
  }, []);
  return <div suppressHydrationWarning>{time || 'Loading...'}</div>;
}
```

### 3. Third-party injected content

```typescript
// ❌ BROKEN: Ad script or extension modifies DOM before React hydrates
// Symptom: React tries to hydrate, finds extra <div> from ad blocker

// ✅ FIXED: Prevent hydration on these elements
function AdSlot() {
  return <div suppressHydrationWarning dangerouslySetInnerHTML={{ __html: '' }} />;
}
```

## THE SUPPRESSHYDRATIONWARNING ESCAPE

```typescript
// ⚠️ Use sparingly — only for truly unavoidable mismatches
// (e.g., third-party injected content)

function DynamicContent() {
  return <div suppressHydrationWarning>{getDynamicValue()}</div>;
}
```

## RENDER FLOW DURING MISMATCH

```
Server renders HTML
  ↓
Client receives HTML
  ↓
ReactDOM.hydrateRoot()
  ↓
Walk DOM tree, match server nodes to virtual tree
  ↓
Compare props on each node
  ↓
MISMATCH FOUND!
  ↓
React deletes the server subtree (innerHTML = '')
  ↓
React re-renders the subtree from scratch
  ↓
Client renders correct version
  ↓
User sees flash/layout shift
```

## PREVENTION MATRIX

| Pattern | Fix | Hydration Safe |
|---|---|---|
| `window.innerWidth` | 🟡 `useEffect` + state | ✅ |
| `new Date()` | 🟡 `useEffect` + state | ✅ |
| localStorage | 🟡 `useEffect` + state | ✅ |
| Random values | 🟡 `useEffect` + state + `suppressHydrationWarning` | ✅ |
| Ad scripts | 🟢 `suppressHydrationWarning` | ✅ |
| CSS-in-JS (styled-components) | 🟢 Use `styled-components` with SSR support | ✅ |
