Component 1: Query Understanding (BERT)

Input Processing

  • Tokenization: Break query into tokens
  • Entity Detection: Identify entities (people, places, things)
  • Intent Classification: What is user looking for?
  • Query Expansion: Add synonyms and related terms
Query: "machine learning" ├─ Tokens: ["machine", "learning"] ├─ Intent: LEARNING (educational) ├─ Entities: TECH_FIELD ├─ Expansion: │ ├─ "ML" │ ├─ "artificial intelligence" │ └─ "deep learning" └─ Output: query_embedding (768D BERT)

Component 2: Candidate Retrieval (Inverted Index)

Inverted Index Structure

  • Format: word → [doc_ids]
  • For query "machine learning":
  • Lookup "machine" → 500M docs
  • Lookup "learning" → 300M docs
  • Intersect → 50M docs with both words
  • Return top 1000 by PageRank
Sharding Strategy: ├─ 1000 shards (by doc_id % 1000) ├─ Query broadcast to all shards in parallel ├─ Each shard searches locally (fits in RAM) ├─ Merge results ├─ Latency: 20ms (parallel execution)

Component 3: Ranking (LambdaMART)

Learning-to-Rank Model

  • Type: Gradient Boosted Decision Trees (GBDT)
  • Training: Millions of human-labeled queries
  • Features: 200+ per document
  • Output: Relevance score [0-100]
Feature Groups: ├─ Query features (50): │ ├─ Query length │ ├─ Intent │ └─ Entity type │ ├─ Document features (100): │ ├─ Title relevance (BM25) │ ├─ Body relevance (BM25) │ ├─ Domain reputation │ └─ Page quality score │ └─ Query-Document features (50): ├─ Title-query overlap ├─ Semantic similarity (BERT) └─ Entity matching

Component 4: Spam & Quality Filtering

Spam Detection

  • Machine-generated content
  • Link spam (artificial PageRank)
  • Thin content
  • Malware sites

Quality Signals

  • Domain reputation
  • User signals (dwell time)
  • Mobile-friendly
  • Page speed

Optimization: Two-Level Ranking

Level 1: Fast Ranker (CPU) ├─ Simple features (BM25, PageRank) ├─ Returns: top 200 ├─ Latency: 30ms Level 2: Slow Ranker (GPU/GBDT) ├─ Complex features (semantic, quality) ├─ Ranks only top 200 ├─ Latency: 50ms Result: 80ms total (vs 100ms) Quality: Comparable (re-ranking improves signal)

Caching Strategy

Hot Queries

Top 1M queries get fully cached results

Hit rate: 60%
Latency: <10ms

Index Sharding Cache

150GB per shard in RAM across fleet

Shards in RAM: 100 of 1000
Hit rate: >95%

Production Challenges

🔴 Failure: Index Shard Down

Symptom: 9 results instead of 10

Solution: 3x replication + automatic failover. Complete request without waiting for failed shard.

🔴 Failure: Ranking Model Degrades

Symptom: CTR drops after model update

Solution: A/B test on 1% first. Rollback in 5 min. Investigate offline.

🚀 Scaling 10x

Current: 100K QPS | Scaled: 1M QPS

Solutions: Model distillation (10x faster) + approximate ranking (rank only top 100)
