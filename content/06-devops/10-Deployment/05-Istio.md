# Istio

Istio is a feature-rich service mesh that uses a sidecar proxy (Envoy) architecture to provide traffic management, security, and observability for microservices. The control plane (istiod) manages configuration and distributes certificates, while Envoy proxies handle all data-plane traffic. Istio is the most widely adopted service mesh for Kubernetes and offers the most comprehensive feature set.

## Overview

Istio deploys an Envoy proxy as a sidecar container alongside each service pod. All incoming and outgoing traffic flows through the Envoy proxy, which enforces routing rules, collects telemetry, and handles mTLS. Istiod (the control plane) manages proxy configuration via xDS protocols, issues certificates via the CA integration (using SPIFFE identities), and aggregates telemetry. Services communicate with each other over mTLS automatically, with no code changes required.

## Key Characteristics

- **Envoy Sidecar**: Envoy is a high-performance L7 proxy written in C++. It supports HTTP/1.1, HTTP/2, gRPC, and TCP. It handles advanced load balancing (locality-aware, circuit breaking, retries, timeouts) and produces detailed telemetry (HTTP metrics, access logs, distributed tracing). In Istio, Envoy terminates mTLS and implements authorization policies at the proxy level.
- **VirtualService**: A routing resource that defines how traffic is directed to a service destination. Supports match conditions (URI, headers, method, port), route weights (for canary deployments), fault injection (delays, aborts), and mirroring (send copy of traffic to a shadow service). VirtualServices decouple traffic routing from service identity.
- **DestinationRule**: Defines policies that apply to traffic after routing. Configures load balancer type (round robin, least request, random, consistent hash), connection pool settings (max connections, connect/request timeouts), outlier detection (circuit breaker thresholds), and TLS settings (mTLS mode: DISABLE, ISTIO_MUTUAL, SIMPLE, MUTUAL).
- **AuthorizationPolicy**: Istio's authorization model for controlling access to services. Supports ALLOW, DENY, and AUDIT actions. Rules match on source identities (principals, namespaces, IP blocks), request properties (paths, methods, headers), and conditions (JWT claims, request metadata). Policies are enforced by Envoy at the proxy level — unauthorized requests never reach the application.
- **PeerAuthentication**: Defines mTLS mode for service-to-service traffic: `PERMISSIVE` (accept both plaintext and mTLS, for migration), `STRICT` (require mTLS), or `DISABLE`. Set at mesh-wide, namespace, or workload granularity. Migration from PERMISSIVE to STRICT is the standard rollout pattern for introducing mTLS to an existing mesh.
- **RequestAuthentication**: Validates JWTs for end-user authentication. Specifies JWKS issuers, token locations (headers, query parameters), and audience validation. Validated claims are forwarded to the application as HTTP headers (X-Forwarded-*, X-Envoy-*), enabling authorization policy enforcement based on JWT claims.
- **Telemetry**: Istio exports rich metrics (request count, duration, size, TCP throughput), distributed tracing (Zipkin, Jaeger, Datadog, OpenTelemetry), and access logs to various backends. Telemetry is generated by Envoy and collected with minimal overhead. Service graphs visualize service dependencies and health.

## Why It Matters

Istio provides the most comprehensive service mesh capabilities for organizations that need advanced traffic management, security policies, and observability. Its Envoy-based architecture is battle-tested at massive scale. Istio's policy model enables fine-grained zero-trust security, and its traffic management supports sophisticated deployment strategies (canary, blue-green, fault injection testing) without application changes.

## Mental Model

Istio is like a smart highway system for microservices. Istiod is the traffic management center that monitors all roads and adjusts traffic lights and speed limits dynamically. Each Envoy proxy is an on-ramp sensor that logs every vehicle entering and leaving, enforces speed limits (rate limits), checks vehicle credentials (mTLS), and can direct some cars to a new highway extension (canary routing). Drivers (services) just follow the road — they don't need to know about the smart system managing their journey.
