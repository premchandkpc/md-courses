🎯 Consistency Architecture

Consistency models from strong to eventual, their trade-offs, and how to choose the right model for each data access pattern.

Flow: Client Write → Primary/Load → Replicas (sync/async) → Read from Replica. Consistency level determined by read-write path configuration.

⚡ Strong Consistency

Read-after-write guarantees. Sync replication, quorum reads (R+W > N). Higher latency but no stale reads.

🔄 Eventual Consistency

Async replication, stale reads possible but converge over time. DNS converges in ~TTL, DynamoDB in ~1s.

🎯 Read-After-Write

Write goes to primary, immediate reads from primary, later reads from replica after ack. Session-level guarantees.

🌀 Conflict Resolution

Last-write-wins, vector clocks, CRDTs, application-level merge. NWR tuning in Cassandra/DynamoDB.

StrongLinearizability: every read sees latest write. Spanner, ZooKeeper, etcd
CausalCausally related operations ordered. DynamoDB, Cosmos DB
EventualReplicas converge without coordination. DNS, S3
Trade-offStrong = higher latency (sync replication, quorum). Eventual = faster writes
