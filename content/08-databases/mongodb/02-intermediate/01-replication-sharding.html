Write Concern Levels

  • w: 1 - Primary only (fast, risky)
  • w: 3 - All 3 replicas (slow, safe)
  • w: "majority" - Half + 1 (balanced)
  • j: true - Journaled (durable)

Read Preference

  • primary - Consistent (default)
  • secondary - Offload reads, eventual consistent
  • nearest - Lowest latency
  • primaryPreferred - Fallback to secondary

Benefits

  • Automatic failover <10s
  • Read scaling (read from secondaries)
  • Backup via secondary
  • Maintenance windows

Monitoring Sharded Clusters

sh.status()

Shows all shards, databases, collections, chunk distribution, balancer status. Run regularly to catch imbalance.

Chunk Distribution

Should be balanced. If Shard1=500 chunks, Shard2=100, Shard3=50 → Bad shard key or new data pattern.

Balancer

Automatic process. Moves chunks to balance. Can be disabled for maintenance window.

Common Sharding Issues & Solutions
Hotspot Shard (90% traffic to 1 shard)
Cause: Bad shard key choice (low cardinality, monotonic, or skewed distribution)
Solution: Reshard with compound key (region + customerId)
Uneven Distribution
Example: Shard1=50M docs, Shard2=40M, Shard3=45M, Shard4=2M
Fix: Reshard with better key or add suffix prefix
New Shard Gets No Data
Check: Balancer running? Chunk size reasonable?
Monitor: sh.getBalancerState(), sh.isBalancerRunning()

Best Practices

Shard Key Design

  • High cardinality (millions of values)
  • Even distribution (avoid skew)
  • Query-aligned (if queries by userId, use userId)
  • NOT monotonic (avoid timestamp)
  • Consider compound keys

Operational

  • Always use replica sets within shards
  • Monitor balancer continuously
  • Backup before and after migration
  • Test with production-like load
  • Plan chunk size strategy

Avoid

  • Changing shard key (very expensive)
  • Low cardinality keys
  • Monotonic sequences (ids, timestamps)
  • Unbalanced distribution
  • Transactions spanning shards (slow)
