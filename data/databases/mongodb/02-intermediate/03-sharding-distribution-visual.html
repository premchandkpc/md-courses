Real-World Scaling Scenarios
Three production examples showing actual shard key choices, chunk distribution, latency metrics, and lessons learned. Each represents different workload characteristics.
📦 Scenario 1: E-Commerce Order Database (500M Orders)
Shard Key: {customer_id, order_date}
Why: Compound key balances even distribution (by customer_id hash) with query efficiency (range on order_date). Most queries are "orders for customer X in date range Y".
Cluster Config: 5 shards (3 replicas each) = 15 nodes
Data Distribution: 100M orders per shard, evenly balanced at 20% ±2% each
Write Throughput: 250K ops/sec (50K per shard × 5)
Query Latency: Targeted: 8ms, Broadcast: 120ms
Monthly Cost: $2,500 (instances + storage + rebalancing)
✓ Key Success: Time-series partitioning (by month) enables archive strategy
🏆 Scenario 2: Gaming Leaderboard (1B Scores)
Shard Key: {game_id, tier}
Why: Games have uneven popularity (some >100M scores, others <1M). Sharding by game_id alone creates hot shards. Adding tier ({bronze, silver, gold, platinum}) further distributes within popular games.
Cluster Config: 3 shards initially, 7 shards after scaling
Data Distribution Challenge: Game A = 40% data, Game B = 35%, Game C-Z = 25% across multiple shards
Solution: Sub-sharding within app layer for Game A (consistent hashing of player_id within shard)
Write Throughput: 500K ops/sec at peak (leaderboard updates concurrent)
Query Latency: Top 100 leaders: 45ms (partial broadcast), Player rank: 12ms
⚠ Challenge: Hot game tier requires monitoring + potential chunk splitting every 2-3 weeks
📊 Scenario 3: Logging & Analytics (10B Events/Day)
Shard Key: {timestamp_bucket, service_id}
Why: Time-series data (all writes to latest day). Without bucket, all writes would go to final shard. Bucketing by hour or 6-hour window distributes across shards. service_id secondary ensures service logs clustered (locality).
Cluster Config: 8 shards, rolling retention (drop logs >90 days)
Write Pattern: 1.2M events/sec ingestion rate distributed across all shards
Query Patterns:
  • Logs for service X, last 24h → targeted + range: 30ms
  • Errors across all services, last hour → broadcast: 200ms (must scan all)
  • Aggregation (error rate by service) → 500ms (merge on primary)
Storage per Shard: ~150GB (rolling), auto-drop oldest chunks when size >200GB
✓ Success: Chunk lifecycle automation (auto-create new buckets, drop old) eliminates manual work
